# PortScanner 端口扫描 — 原理说明

## 概述

端口扫描是网络渗透测试的第一步。通过向目标主机的各端口发送特定类型的数据包，根据响应判断端口状态及运行服务，从而确定攻击面。

---

## 1. TCP 三次握手与扫描原理

正常 TCP 连接建立需要三次握手：

```
客户端                        服务器
  │─── SYN ───────────────→│     (1) 客户端发送 SYN)
  │←── SYN+ACK ────────────│     (2) 服务器回复 SYN+ACK)
  │─── ACK ───────────────→│     (3) 客户端发送 ACK)
  │         连接建立         │
```

不同扫描技术通过**不完成完整握手**或**发送非标准标志位**来推断端口状态，同时尽可能隐蔽。

---

## 2. 扫描类型对比

### TCP Connect Scan（全连接扫描）

- 完成完整三次握手，调用 `connect()` 系统调用
- 优点：无需 root 权限，兼容性好
- 缺点：速度慢，连接完成时目标应用会记录日志，最容易被检测

### SYN Scan（半开扫描，默认推荐）

- 只发送 SYN，收到 SYN+ACK 后立即发 RST 断开，不完成握手

```
扫描端                        目标
  │─── SYN ───────────────→│     [open] 回复 SYN+ACK
  │─── RST ───────────────→│     扫描端立即 RST 断开
```

- **隐蔽性**：连接未建立，应用层无日志（大多数服务只记录完整连接）
- **速度**：无需等待三次握手完成，更快
- **缺点**：需要 root/raw socket 权限构造原始数据包

### FIN / Xmas / Null Scan（隐蔽扫描）

利用 RFC 793 规定：**对关闭端口，收到不含 SYN/RST/ACK 的包必须回复 RST**；而开放端口应忽略。

| 扫描类型 | 标志位 | open 响应 | closed 响应 |
|:---------|:-------|:----------|:------------|
| **FIN** | FIN | 无响应 | RST |
| **Xmas** | FIN+PSH+URG | 无响应 | RST |
| **Null** | 无标志位 | 无响应 | RST |

**局限**：Windows 系统不遵循 RFC，无论端口是否开放都回复 RST，对这些系统无效。

## 3. 端口状态定义

| 状态 | 含义 | 判断依据 |
|:-----|:-----|:---------|
| **open** | 端口有应用监听，接受连接 | SYN 扫描收到 SYN+ACK |
| **closed** | 端口无应用监听，但主机可达 | SYN 扫描收到 RST |
| **filtered** | 防火墙/过滤设备阻止了探测包 | 无响应或收到 ICMP unreachable |

## 4. 服务探测（Banner Grabbing）

发现 open 端口后，发送应用层探测包获取服务指纹：

```
端口 80  → 发送 "GET / HTTP/1.0\r\n\r\n" → 解析 Banner 中的 Server/版本
端口 22  → 发送空数据 → 读取 SSH Banner "SSH-2.0-OpenSSH_8.9"
端口 21  → 发送空数据 → 读取 FTP Banner "vsFTPd 3.0.3"
端口 3306→ 读取 MySQL 握手包 → 解析版本号
```

服务探测的匹配逻辑：**请求特定 probe → 正则匹配响应 → 确定服务名+版本**（类似 nmap-service-probes 数据库）。

## 防御建议

| 措施 | 说明 |
|:-----|:-----|
| **IDS/IPS 部署** | Snort/Suricata 检测异常扫描模式（如短时间大量 SYN 到不同端口） |
| **防火墙配置** | 默认拒绝入站，仅开放必要端口；关闭不使用的服务 |
| **端口敲门（Port Knocking） | 需按特定顺序访问一组端口后才开放真正的服务端口 |
| **SYN Cookie** | 防止 SYN Flood 攻击占满半连接队列 |
| **关闭 Banner 信息** | 移除服务版本号等指纹信息，增加攻击者探测难度 |
| **速率限制** | iptables 限制单 IP 单位时间新连接数 |
